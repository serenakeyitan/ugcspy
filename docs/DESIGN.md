## What actually shipped (delta vs original spec)

The original spec assumed an Anthropic API key for hook extraction + brief generation, and ScrapeCreators (paid) as the default data provider. Both got pivoted during the build:

- **No Anthropic API key.** Brief generation moved into the Claude Code plugin and runs via the user's existing Claude Code subscription. Standalone CLI uses caption-only hook extraction (free, deterministic). See commit 03ba74d.
- **Free TikTok-OSS provider as default.** Wraps davidteather/TikTok-Api via a Python subprocess. ScrapeCreators is now an optional upgrade for users who need IG Reels coverage. See commit 5ca3883.
- **Hashtag-mode search is the default for plain-word queries** (`ugcspy search befreed`). The original spec only had `@handle` search (the brand's own posts). Real BigSpy-for-UGC use case is *finding third-party creators promoting a brand*, which requires hashtag search. See commit 5ed4073.
- **Ranking is by views (reach), not engagement rate.** Engagement-rate ranking inflated tiny videos above genuinely viral ones. Now matches BigSpy's actual behavior. See commit f0284a5.
- **Four-pass coverage strategy + repeat-querying + bounded parallelism for hashtag search.** Single-hashtag returns ~140-200 per call. The shipped CLI runs (0) user-search seeding, (1) hashtag fetch, (2) campaign-code discovery, (3) seed-creator walk; each hashtag is also repeat-queried up to 3 times until a round adds fewer than 5 new videos. Within each pass, fetches run concurrently with concurrency=12 by default (verified empirically — 4 parallel = 2.97x speedup over sequential, 12 parallel = 1.6x speedup over 8, all zero errors on fresh IPs). Configurable via `UGCSPY_CONCURRENCY` env var; users with `MS_TOKEN` cookies can safely push to 16-24. Wall time ~62-68s for an active brand; corpus 410-444 videos; max view ceiling 335K. See commits 2093f7e, 64e492a, 8a7cd90, 85c0cc2, ccfa543, e62c560.
- **Watch + daemon (alerts) shipped but moved to "Optional" in the docs** — not part of the demo flow. Search + fork is the core. See commit f0284a5.
- **Browser-free discovery rebuilt: wide-collect → signal-rank → snowball → yt-dlp coverage.** The Chromium discovery passes (0-2) crash ("No valid sessions found") and hang to timeout, silently shrinking the roster. Replaced with a pure-HTTP tikwm pipeline (Chromium now OFF by default; `UGCSPY_USE_CHROMIUM=1` re-enables it as an extra source on a residential IP). Three lessons baked in:
  1. **Discovery must be WIDE, precision belongs downstream.** The old code filtered creators at discovery by whether their *one surfaced* video's title carried the brand — dropping creators whose search-result clip didn't name the brand even though their catalog is full of brand UGC. This shrank a ~1400-candidate brand search down to ~38. Fix: discovery collects every surfaced creator; the per-video brand filter runs only in the yt-dlp coverage pass (which walks each creator's full catalog). Measured on BeFreed: **38 → 1403 candidate creators.**
  2. **Rank candidates by signal, then walk by rank.** Wide discovery surfaces a long noise tail from broad keywords. Each candidate gets a score (+1 per surfacing, +2 if the surfaced title carries the brand, +3 for hashtag-sourced, +2× for each known brand creator who follows them). The coverage pass walks highest-signal first, so under the seed cap the genuine brand creators get crawled and noise falls below the cut.
  3. **Following-graph snowball is the biggest long-tail source.** Brand-UGC creators mutually-follow as a tight collective, so walking who the high-signal seeds *follow* (tikwm `/api/user/info` → `/api/user/following`, depth-1) surfaces creators search never ranks high enough to return (e.g. `@bobby/@eilisa/@lance/@paige.befreed`). Knobs: `UGCSPY_SNOWBALL_PAGES` (default 1, 0 disables). Walk concurrency raised 4→8 (`UGCSPY_WALK_CONCURRENCY`) now that the roster is ~200 not ~38. Estimated combined ceiling ~600-900 creators for a mid-size brand; beyond that needs residential proxies (deferred). Sound-id discovery and wider keyword expansion were evaluated and skipped (near-zero marginal value / noise).

## Known limits of the free path (documented honestly)

- `User.following` is not exposed by TikTokApi v7 — TikTok gates following lists behind auth. **Worked around** via the tikwm relay (`/api/user/following`, browser-free), which powers the following-graph snowball discovery source above — so we now CAN enumerate the brand-creator follow-collective without auth. (TikTokApi pass-0 user-search remains as a Chromium-only fallback.)
- Single-hashtag responses cap at ~140-200 per call. TikTok rotates the feed slightly between calls, so repeat-querying works (~+39% unique videos after 5 rounds), with diminishing returns by round 3-4. The CLI now repeat-queries up to 3 rounds per hashtag with a saturation cutoff.
- `User.liked` returns 0 for most accounts — TikTok hides likes by default.
- `Search.videos` is not exposed by TikTokApi (only `Search.users` works in v7), so the
  TikTokApi-based modes structurally cannot do keyword/niche discovery. **Resolved** by
  adding a `keyword` mode that goes through the free **tikwm** relay (`/api/feed/search`)
  over plain HTTP — no TikTokApi, no Chromium, no key. This is the only path that finds
  untagged niche/competitor UGC (videos that don't hashtag a brand). tikwm is an
  unofficial third-party relay (no SLA, ToS gray area) so the keyword path fails soft to
  an empty list and is independent of the TikTokApi user/hashtag modes — one source dying
  doesn't break the others. The official TikTok Research / Creative Center API is the
  durable-but-gated upgrade path.

For the auth-gated gaps above, the upgrade is paid ScrapeCreators (handles auth-required
endpoints).

## Problem Statement

Brand social media managers (SMMs) are already paying $300-1000/mo for tools like Trendpop, Pentos, Exolyt, Sprout Social, and Dash Social. None of them solve the specific job they actually have: **"track my competitors' organic UGC the way BigSpy tracks competitors' ads."** Existing tools are either bloated SMM platforms where competitor tracking is a side-feature buried in scheduling/inbox/reporting, or single-platform analytics that surface raw stats without the BigSpy form factor (search-first, fast, cheap, self-serve, no sales call).

The user has watched 3+ brand SMMs struggle with this in person. They are paying for tools that do not solve the core need.

## Demand Evidence

- User has personally observed 3+ brand SMMs paying for Trendpop / Pentos / Exolyt and still failing to do "track competitor X's UGC."
- Industry context: creator marketing is $22.2B; 93% of brands say UGC outperforms branded content; SMM tools market is mature with budget unlocked at $79-750/seat-month.
- Behavioral signal: SMMs are *already paying* for adjacent tools — buying pattern established. They aren't being asked to find new budget.

**Gap:** No one in the landscape ships a BigSpy-shaped product (search-first, $29-99/mo, self-serve, no sales call) for organic UGC. Everyone builds platforms.

## Status Quo

A brand SMM's current workflow:
1. Subscribe to Trendpop/Pentos for $300-500/mo (TikTok-only or IG-only).
2. Manually scroll competitor accounts in the native app and screenshot anything interesting.
3. Maintain a Notion or spreadsheet of "what worked for [competitor]."
4. Forward links to creators in Slack with "make something like this."
5. Lose track of which competitor videos are *currently* outperforming, because there's no monitoring/alert layer — only point-in-time browsing.

Cost: 3-5 hours/week of SMM time + $300-500/mo on tools that don't do the job.

## Target User & Narrowest Wedge

**Form-factor reality check:** because V1 is a CLI requiring `npm install -g`, an Anthropic API key, and a scraper API key, the *immediate* user is technical. Non-technical SMMs get this via a future hosted tier ("Cloud" — out of V1).

**V1 ICP (technical operator):** Founder/engineer at a DTC consumer brand who runs/helps run social. Or: dev/ops at a social-marketing agency serving DTC brands. They are comfortable in a terminal, want a fast tool that beats their SMM's $300/mo subscription, and value that it's open source + scriptable + agent-native (drops into their Claude Code / Cursor / n8n workflows).

**V1 ICP secondary:** Power-user marketers who already use Claude Code (a real and growing demographic — agentic workflows are pulling marketing into the terminal).

**V2 ICP (Cloud tier):** Brand SMMs who can't run a CLI. Hosted version where we run the daemon, manage API keys, and they paste a Slack webhook into a web form. Defer to post-hackathon.

**Out of V1:** Solo creators (no budget, free tools exist).

**Narrowest wedge:** Three commands. `search` ranks a competitor's last 30 days of TikTok + IG Reels. `watch` alerts to Slack on threshold breach. `fork` turns any video into a creator brief. Open source. Agent-native. Faster + cheaper + more scriptable than Trendpop/Pentos for technical users.

## Premises

1. **ICP is brand SMMs (and agencies serving them), not creators.** [agreed]
2. **Wedge is form factor; detection quality is the non-negotiable floor.** [revised after Codex challenge] Form factor (CLI + agent-native + open source) is the differentiator vs. SaaS dashboards. But detection quality is the floor: if alerts are noisy or coverage is patchy, the CLI doesn't matter. We commit to measurable bars on detection before V1 ships (see Day 1 and Day 4 acceptance criteria).
3. **Killer feature is "type a competitor → ranked organic UGC + AI-extracted hooks" + alerts on threshold breach + one-click fork to creator brief.** [agreed, with user-added "alerts" and "fork"]
4. **TikTok + Instagram Reels are V1. YouTube Shorts + X are V2.** [agreed]
5. **Distribution is PLG: ProductHunt + Twitter + SMM communities. No outbound sales, no paid ads.** [agreed]
6. **Data acquisition is the hardest part. V1 uses third-party data providers (ScrapeCreators / Apify / Bright Data). Build own scrapers later as scale demands.** [agreed]

## Approaches Considered

### Approach A: BigSpy clone, exact form factor (search only)
Tightest wedge, ships in 3 days, no alerts/fork. Easiest demo, lowest differentiation. Effort: S. Risk: Low.

### Approach B: Competitor Radar (search + alerts + fork) [SELECTED]
Full product loop: discover → monitor → act. 5-7 day build. Three "oh shit" moments in demo (search, live alert, fork-to-brief). Effort: M. Risk: Medium.

### Approach C: AI Brand Watcher (agent-first)

## Recommended Approach

**Approach B — Competitor Radar (CLI + Claude Code skill).** Three commands in V1:

### Architecture: CLI engine, skill wrapper

- **CLI** is the real product: a TypeScript/Bun binary distributed via `npm install -g competitor-radar` and `brew install competitor-radar`. It owns config (`~/.competitor-radar/config.json`), local SQLite cache (`~/.competitor-radar/db.sqlite`), API keys, scraper integration, and AI calls.
- **Claude Code skill** is a thin Markdown wrapper at `.claude/skills/competitor-radar/SKILL.md` that shells out to the CLI. Skill invocation: user types `/competitor-radar search @glossier` inside Claude Code → skill runs `competitor-radar search @glossier --json` → Claude renders the JSON nicely in chat.
- **Why both:** CLI works in any terminal (judges, agencies, power-user SMMs). Skill makes it free agent-native (Cursor, Claude Code users get it for $0 marginal effort).

### Three V1 commands

1. **`competitor-radar search <handle>`** — Fetches the handle's last 30 days of TikTok + IG Reels, ranks by engagement, prints a table to stdout (or `--json` for machine consumption). Each row: rank, platform, posted_at, views, engagement_rate, format_tag, hook_text, url. Default 20 rows, `--limit N` to change.
2. **`competitor-radar watch <handle> --slack-webhook <url> --threshold 2x`** — Adds the handle to a watch list (stored in `~/.competitor-radar/watches.json`). A separate `competitor-radar daemon` command (or a `cron` entry, or a GitHub Action the user installs) polls every 6h and posts to the user's Slack webhook when a video crosses 2x the trailing-30-day-median views-at-24h. **Cold-start:** same rule as before — `warming_up` until 7 days + N≥5 videos. Slack webhook only in V1; no daemon hosting from us, the user runs `competitor-radar daemon` themselves or sets up cron / GitHub Actions cron.
3. **`competitor-radar fork <video-url-or-id>`** — Sonnet 4.6 generates a creator brief and writes it to `~/.competitor-radar/briefs/brief-<id>.md` (default). `--out <path>` to override location, `--copy` to put on clipboard instead. Brief includes: hook variations, format notes, suggested b-roll, length, CTA.

### Hook + format extraction (unchanged from prior section)

Same multi-source hook extraction (caption → overlay vision → Whisper fallback) and same 10-tag closed-list format taxonomy. Now runs locally inside the CLI, calling Anthropic API directly with the user's API key.

### Config and credentials

- First run: `competitor-radar init` walks user through pasting their (a) Anthropic API key, (b) chosen scraper provider's API key (default: ScrapeCreators), (c) optional Slack webhook URL.
- Stored in `~/.competitor-radar/config.json` with `chmod 600`. No telemetry, no phone-home. Open source.
- **Cost model for V1 user:** they pay scraper provider and Anthropic directly. **Real per-handle cost must be measured in Day 0 spike** — current `~$0.001-0.01/handle/day` is an estimate, not a quote. Day 0 deliverable includes a one-page cost report at "10 handles polled every 6h" so we can quote users honestly.
- **Monetization, V0:** open-source CLI is free. **Pre-launch a "Cloud" waitlist on Day 7** with a Stripe payment link ($29/mo "managed daemon — we run the cron, you paste a Slack webhook"). This is the path to the Month-1 paid customer success criterion. If the design partners install the CLI happily, great; if they say "I want this but not the install," they convert to Cloud waitlist. Either signal is valuable.
- **Monetization, V1+:** Cloud (hosted daemon for non-technical SMMs, $29/mo), agency tier (multi-brand dashboards, $99/mo), or licensed premium data. Pick after design-partner conversations, not now.

### Hook Extraction (multi-source, caption-primary)

TikTok audio is dominated by music + voiceover. Whisper alone is unreliable (40%+ WER on music-overlaid speech). V1 ranks sources:

1. **Caption text** (free, always available, often the actual hook).
2. **On-screen overlay text** — vision model (Claude Sonnet 4.6 with image input) reads frame-0 text. Most TikToks have a hook in overlay.
3. **Whisper transcript of first 5 seconds** — supplemental only. If caption + overlay both empty, fall back to Whisper.

Hook field stores `{source: "caption" | "overlay" | "whisper", text: string, confidence: number}`.

### Format Taxonomy (closed list — 10 tags)

Classifier prompt asks Sonnet 4.6 to pick exactly one of:
`GRWM` (get ready with me), `POV`, `talking_head`, `product_demo`, `unboxing`, `tutorial`, `before_after`, `voiceover_broll` (no on-camera person), `duet_stitch`, `other`.

Stored as enum. Surfaced as filter chips in search UI.

### Tech Stack

- **Runtime:** Bun (fast install, single binary, native TypeScript). Fallback to Node 20+ if a user's environment can't install Bun.
- **CLI framework:** [`commander`](https://www.npmjs.com/package/commander) for arg parsing. [`ink`](https://www.npmjs.com/package/ink) or `chalk` + `cli-table3` for pretty stdout.
- **Local storage:** SQLite via `better-sqlite3` for cached video data and watch state. Lives in `~/.competitor-radar/db.sqlite`.
- **Data provider:** V1 default is `tiktok-oss` — a free, OSS path wrapping davidteather/TikTok-Api (6.3k stars, v7.3.3 shipped April 2026) via a Python subprocess bridge. Covers TikTok only. For Instagram Reels, user picks `scrapecreators` (paid) — no production-grade free IG scraper exists right now. Day 0 spike (`scrapecreators` vs `apify` vs `bright_data`) becomes a *post-V1* concern, only when a paying user hits the OSS rate limits or asks for IG.
- **AI:** Claude Haiku 4.5 for format tagging + hook extraction. Claude Sonnet 4.6 (with image input) for overlay-text vision and brief generation. OpenAI Whisper API for fallback transcription only when caption + overlay both empty. (Anthropic does not expose a Whisper-equivalent.)
- **Background work:** No daemon hosting from us in V1. **Recommended path: GitHub Actions cron** — `competitor-radar export-gha > .github/workflows/radar.yml` writes a workflow file that runs every 6h on GitHub-hosted runners with API keys stored as GitHub Secrets. Zero local infra, zero ongoing cost (within free tier). Fallback: `competitor-radar daemon` for users who already run a server. Cron / `tmux` / `pm2` are documented but de-emphasized.
- **Auth:** none. Local CLI. No accounts, no Clerk.
- **Distribution:** `npm install -g competitor-radar` (primary), `brew install competitor-radar` (Homebrew tap), `bun install -g competitor-radar`. Open source on GitHub.
- **Claude Code integration:** Ship as a **Claude Code plugin** so users get explicit slash commands (`/competitor-radar-search`, `/competitor-radar-watch`, `/competitor-radar-fork`). Plugin includes a SKILL.md (with required YAML frontmatter — `name`, `description`) plus slash-command definitions that shell out to the CLI binary. Distributed in the same GitHub repo. The skill description triggers on intent ("when user wants to research competitor UGC") so it also works without explicit slash invocation.

### Schema notes (SQLite, local)

- `competitors`: `(id, handle, platform, added_at)`. `(handle, platform)` unique.
- `videos`: `(id, competitor_id, platform, external_id, posted_at, fetched_at, caption, thumbnail_url, view_count, like_count, comment_count, share_count, hook_source, hook_text, hook_confidence, format_tag, raw_metrics_json)`. `(platform, external_id)` unique.
- `watches`: `(id, competitor_id, slack_webhook_url, threshold_multiplier, state ['warming_up'|'active'], created_at)`.
- `alerts_fired`: `(id, video_id, watch_id, fired_at)` to prevent duplicate Slack pings.
- **Refresh strategy:** when daemon runs (or `search` is called), fetch latest videos for the handle, re-fetch metrics for videos posted in last 7 days only. Older videos frozen. Soft-delete on 404.

### Build Order (5-7 days)

**Day 0 (gate — must pass before Day 1):**
- 2-hour spike: test ScrapeCreators, Apify, Bright Data on `@glossier` TikTok + IG Reels. Compare freshness, completeness, cost. Pick one. If all three fail on IG: V1 is TikTok-only.
- Bun project init: `bun init`, `commander` + `better-sqlite3` + `@anthropic-ai/sdk` installed. SQLite schema migration runs. **Acceptance:** `competitor-radar --help` prints something.
- **Kick off baseline ingestion for the 20-brand demo competitor set in parallel** (just the provider's ingestion path, no CLI surface yet). Reason: Day 7 demo's recorded alert needs a real 7-day baseline — collection has to start Day 0.

**Day 1:** `competitor-radar init` (interactive setup writes config) + `competitor-radar search <handle>` (fetches → SQLite → pretty stdout table). **Acceptance:** (a) `competitor-radar search @glossier` returns ranked videos in stdout in <5s on second call; (b) **detection-quality bar:** for each of 5 demo brands, ingested video count is ≥95% of the actual count posted in the last 30 days, verified by manual scroll of the brand's profile. If <95%, fix the scraper integration or switch providers before moving to Day 2; (c) ingestion path is parameterized so Day 6 can pre-load 20 brands via a one-line config.

**Day 2:** Hook extraction (caption → overlay vision via Claude Sonnet 4.6 image input → Whisper fallback) + format tagger (Haiku 4.5, closed list of 10). **Acceptance:** every row in `competitor-radar search` output has hook_text + format_tag populated.

**Day 3:** Output polish + filters. `--sort engagement|recency`, `--format GRWM,POV`, `--platform tiktok|ig`, `--json` flag. **Acceptance:** stdout looks demo-quality (color, alignment, hook truncation, clickable URLs in iTerm/Warp). JSON output validates against documented schema.

**Day 4:** `competitor-radar watch <handle>` + `competitor-radar daemon` (polls every 6h, fires Slack on threshold breach, respects warmup). **Acceptance (basic):** `watch @brandX --slack-webhook https://...`, manually insert a synthetic high-view row, run daemon once, see real Slack ping fire. **Acceptance (detection-quality bar):** build a hand-labeled set of 50 real breakouts from the demo brands' last 90 days (videos that crossed 2x median in 24h, by manual review). Run the alert pipeline on that backtest. **Recall ≥80%** (alert fires on 40+ of 50 real breakouts). **FPR ≤20%** (in the same window, ≤20% of fired alerts are on videos that didn't sustain the breakout). If either bar fails, tune threshold or denominator before Day 5. If still failing after one tune cycle, escalate — V1 doesn't ship without these bars.

**Day 5 (BUFFER + Fork):** `competitor-radar fork <video-id-or-url>` writes `./brief-<id>.md` (or `--copy` to clipboard) using Sonnet 4.6. Day 5 absorbs slippage if Days 1-4 ran late. **If Fork is cut**, ship without it and say so publicly — no fake demos.

**Day 6:** Pre-load the 20-brand demo set into SQLite. Write the `.claude/skills/competitor-radar/SKILL.md` wrapper. End-to-end polish. Fix the 10 bugs.

**Day 7:** Demo prep. Demo flow: (a) terminal: `competitor-radar search @glossier` → ranked table in <3s (live), (b) `competitor-radar fork <picked video>` → brief.md opens in editor (live), (c) play a 30-second screen recording of a real Slack alert that fired in test env on Day 6, narrated honestly as "this fired yesterday on @brandX," (d) bonus: open Claude Code, type `/competitor-radar search @rarebeauty`, show the agent-native form factor as the kicker. Record 90-sec backup video in case wifi dies.

## Open Questions

1. **Data provider — ScrapeCreators vs Apify vs Bright Data.** **GATE: Day 0 spike picks one before Day 1 starts.** Test the same handle (`@glossier`) across all three. Compare freshness, completeness on IG Reels specifically, cost-per-call. If all three fail on IG, V1 is TikTok-only.
2. **Format taxonomy validity.** The 10-tag closed list above is best-guess. After Day 2, sample 50 classified videos and check: are all 10 tags actually used? Is anything getting forced into `other` >20%? Tune before Day 6.
3. **Alert false-positive rate.** "2x trailing-30-day median views-at-24h" is the spec. Risk: posting peaks (Sunday 5pm), holidays, viral one-offs. Day 4 acceptance includes a synthetic-data backtest on 1 competitor's 30 days. Tune threshold if FPR > 30%.
4. **Pricing.** $29/$99 is unvalidated. Test against 5 SMM design partners post-hackathon before locking.
5. **Legal/ToS exposure.** TikTok and Meta both prohibit scraping; the data provider shifts liability but does not eliminate it. **Plan:** stay on third-party providers in V1; if a cease-and-desist arrives, migrate to (a) official TikTok Research API where eligible, or (b) drop affected platform until alternative is found. Add Terms-of-Service line: "Data sourced from public posts via third-party providers; we do not store or rehost video content."

## Success Criteria

### Hackathon demo
- Judges name any brand **from the pre-loaded set of 20** (state this explicitly: "we ingested 20 DTC brands; pick one"); `competitor-radar search @brand` returns 30 ranked organic videos with hooks in <3 seconds (live, in terminal).
- `competitor-radar fork <video-id>` generates a believable creator brief in <5 seconds (live, brief.md path printed; open in editor on stage).
- A 30-second screen recording of a real Slack alert from Day 6 plays. If the 7-day baseline didn't produce a real 2x breach in time, **honestly script it as a backtested alert** ("we replayed last week's data through the alert pipeline; here's what would have fired") rather than fake a live one.
- Kicker: open Claude Code, run `/competitor-radar-search @brand` via the plugin — same data, agent-native form factor.
- Output polish — terminal looks intentional. Color, alignment, sane defaults.

### Post-hackathon (real startup)
- 5 brand SMMs tested it within 2 weeks of hackathon.
- 3 of the 5 said "I would pay for this" or "when can I have access."
- 1 paid before the end of month 1. (Per Q1 demand evidence rule: paying behavior, not interest.)
